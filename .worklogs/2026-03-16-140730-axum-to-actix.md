# Migrate shedul3r inbound API: Axum → actix-web, rename api → rest

**Date:** 2026-03-16 14:07
**Scope:** apps/shedul3r/crates/adapters/inbound/rest/ (formerly api/), apps/shedul3r/Cargo.toml, apps/shedul3r/deny.toml

## Summary

Renamed the `api` inbound adapter crate to `rest` and migrated from Axum to actix-web. All 41 workspace tests pass. Prerequisite for adding MCP transport as a second inbound adapter.

## Context & Problem

We need to add an MCP (Model Context Protocol) transport to shedul3r so Claude Code can call it directly as an MCP tool. The official Rust MCP SDK (`rmcp`) has an actix-web transport crate (`rmcp-actix-web`). Rather than mixing Axum and actix-web in one workspace, we migrate the existing HTTP adapter to actix-web first.

## Decisions Made

### Rename api → rest
- **Chose:** Rename crate to `rest` to distinguish from the upcoming `mcp` crate
- **Why:** Both are inbound adapters — naming them by transport protocol is clearer

### actix-web 4.x
- **Chose:** actix-web 4 with actix-multipart 0.7
- **Why:** Stable, mature, official rmcp transport support via rmcp-actix-web

### Auth middleware as Transform/Service pair
- **Chose:** Full `Transform`/`Service` middleware implementation instead of closure
- **Why:** Idiomatic actix-web pattern, cleaner than wrap_fn for bearer token auth

### Bundle download uses in-memory read
- **Chose:** `tokio::fs::read` into memory instead of streaming via tokio-util
- **Why:** Simpler without tokio-util dependency; bundles are small enough

## Architectural Notes

- `state.rs` and `cli.rs` needed no changes — they're framework-agnostic
- deny.toml swapped: banned `actix-web` → banned `axum` (only one HTTP framework allowed)
- `#[allow(clippy::future_not_send)]` needed on bundle upload handler because actix-web's `Multipart`/`Field` are intentionally `!Send` (single-threaded workers)

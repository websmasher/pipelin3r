# Add async task API for remote execution

**Date:** 2026-03-18 00:53
**Scope:** apps/shedul3r/ (async endpoints + store), packages/shedul3r-rs-sdk/ (polling client), packages/pipelin3r/ (remote execution)

## Summary

Added async task submission + polling to shedul3r to support remote execution behind proxies (Railway, Cloudflare) that kill long HTTP connections. Client submits task, gets ID immediately, polls for result.

## Context & Problem

Remote execution through Railway's edge proxy times out after ~2 minutes (Cloudflare 524). Agent tasks run 5-15 minutes. The synchronous API blocks the HTTP connection for the entire duration, which no proxy supports.

## Implementation

### shedul3r server
- `POST /api/tasks/async` — accepts TaskRequest, spawns background execution, returns 202 with task_id
- `GET /api/tasks/async/{id}` — returns running/completed/failed status
- `AsyncTaskStore` — in-memory BTreeMap behind parking_lot::RwLock
- Background reaper every 60s, TTL 10 minutes based on completion time (not creation time)

### SDK
- `submit_task_async()` — POST, returns task_id
- `get_task_status()` — GET, returns status
- `submit_task_poll()` — submit + poll loop with max_poll_duration timeout, transient error retry (3 attempts on 5xx)

### pipelin3r
- `execute_with_work_dir` uses `submit_task_poll` when remote, `submit_task` when local

## Bugs Found and Fixed (via adversarial testing)

1. SDK path mismatch — was calling `/api/tasks/{id}` instead of `/api/tasks/async/{id}`
2. No poll timeout — infinite loop if server never completes. Fixed: max_poll_duration deadline
3. No HTTP status check on submit_task_async — 500 with valid JSON accepted as success. Fixed: check is_success()
4. No HTTP status check on get_task_status for non-404 — Fixed: return error on non-2xx
5. No retry on transient errors — single 500 killed polling loop. Fixed: retry up to 3x on 5xx
6. TTL based on creation time — long tasks had shrinking poll window. Fixed: TTL from completion time

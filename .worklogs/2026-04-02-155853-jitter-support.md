# Jitter Support for limit3r

## Summary

Added configurable jitter to all timed components in limit3r (RetryExecutor, RateLimiter, CircuitBreaker). Uses full jitter formula `[base*(1-j), base*(1+j)]` with `rand 0.9`. Defaults to 0.0 (no jitter) for backward compatibility.

## Decisions

- **Full jitter over decorrelated jitter** — simpler, predictable bounds, sufficient for both anti-bot evasion and thundering-herd prevention.
- **Jitter applied before max_delay clamp in retry** — so `max_delay` remains a hard ceiling on actual sleep duration. Initial implementation applied jitter after clamping (contract violation found by adversarial testing).
- **CircuitBreaker jitter computed once at open time** — stored in `jittered_wait` field, reused if circuit re-opens after failed probe (config not available in `record_outcome`). Cleared on close; each new open cycle gets a fresh sample.
- **Rate limiter jitter capped at deadline** — `jittered.min(time_to_deadline)` prevents timeout overshoot. Uses `saturating_duration_since` to avoid panic on expired windows.
- **BulkheadConfig excluded** — bulkhead uses timeouts, not delays. Jitter on a timeout makes no sense.
- **NaN/Infinity defense in depth** — validation rejects non-finite jitter_factor; `apply_jitter` also guards internally with `is_finite()` check.
- **Duration::MAX clamp uses 1e18** — `Duration::MAX.as_secs_f64()` rounds up past `u64::MAX` due to f64 precision loss, causing `from_secs_f64` to panic.

## Key Files

- `packages/limit3r/src/jitter.rs` — new module, `apply_jitter()` utility
- `packages/limit3r/src/config/mod.rs` — `jitter_factor` field + validation on 3 configs
- `packages/limit3r/src/retry/mod.rs` — jitter before max_delay clamp in `compute_delay()`
- `packages/limit3r/src/rate_limiter/mod.rs` — jitter + deadline cap in `acquire_permission()`
- `packages/limit3r/src/circuit_breaker/mod.rs` — `jittered_wait` in `CircuitState`, `ensure_and_evict()` extraction
- `deny.toml` — skip entries for `getrandom@0.3.4` and `r-efi@5.3.0` (rand dep chain)

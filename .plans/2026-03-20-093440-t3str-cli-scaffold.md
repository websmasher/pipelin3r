# t3str — Test extraction and execution CLI

**Date:** 2026-03-20 09:34
**Task:** Build a Rust CLI tool that discovers, filters, and runs tests for multi-language repos. Replaces the embedded-Python-in-Rust approach in the websmasher pipeline.

## Goal

A standalone Rust CLI binary (`t3str`) with two subcommands:
- `t3str extract` — AST-based test discovery and filtering (tree-sitter). Given a repo, language, and topic, returns which test files/functions are relevant.
- `t3str run` — test execution and structured output parsing. Given a repo, language, and optional filter, runs tests and returns standardized JSON results.

The binary ships in the worker Docker image. The pipeline calls it instead of generating Python scripts.

## Input Information

- Current pipeline embeds Python scripts inside Rust `format!()` macros (s06_run_parsers.rs) — source of most bugs
- Grep and regex are banned — use proper parsers (tree-sitter for AST, quick-xml/serde_json for output)
- Must handle 9 languages: Rust, Python, Go, JavaScript, PHP, C#, Ruby, Java, Elixir
- Structured test output formats per language:
  - Python: JUnit XML (pytest --junitxml)
  - Go: JSON (go test -json)
  - Rust: text (cargo test), future: JSON unstable
  - JavaScript: JSON (jest --json), or mocha text
  - PHP: JUnit XML (phpunit --log-junit), or Nette Tester console
  - C#: stdout Passed/Failed lines (TRX is unreliable — one file per project)
  - Ruby: JSON (rspec --format json)
  - Java: JUnit XML (maven surefire)
  - Elixir: text (mix test)
- Each language has isolated env: .venv (Python), .gopath (Go), .cargo (Rust), .bundle (Ruby), node_modules (JS)
- shedul3r worker has all language runtimes installed

### Dependency chain (enforced by Cargo)

```
t3str-cli → t3str-commands → t3str-discovery-port → t3str-domain-types
                ↑                    ↑
           t3str-extract ────────────┘
           t3str-run ────────────────┘
```

| Crate | Depends on | Cannot depend on |
|-------|-----------|-----------------|
| t3str-domain-types | nothing internal | everything |
| t3str-discovery-port | domain-types | commands, extract, run, cli |
| t3str-commands | domain-types, discovery-port | extract, run, cli |
| t3str-extract | domain-types, discovery-port | commands, run, cli |
| t3str-run | domain-types, discovery-port | commands, extract, cli |
| t3str-cli | everything | — (composition root) |

### Key dependencies (external)

| Crate | Purpose |
|-------|---------|
| tree-sitter + language grammars | AST parsing for test discovery |
| quick-xml | JUnit XML / TRX parsing |
| serde + serde_json | JSON test output parsing |
| clap | CLI argument parsing |
| thiserror | Typed error enums |
| tokio | Async process execution |
| tracing | Structured logging |

### Key decisions

- **tree-sitter over regex**: AST-based test discovery finds test functions by structure, not pattern matching. No false positives from comments or string literals.
  - Alternative: regex/grep — banned, fragile, caused repeated pipeline failures
  - Alternative: language-specific parsers (syn for Rust, etc.) — too many to maintain, tree-sitter unifies them

- **Separate parsers/ module in run crate**: JUnit XML parsing is shared by Python, PHP, Java, C#. Extract into reusable parsers rather than duplicating per language module.

- **Single binary, two subcommands**: One artifact to ship to the worker. Simpler Docker image management.
  - Alternative: two separate binaries — more operational complexity for no benefit

- **Process execution via std::process::Command**: t3str runs ON the worker alongside the language runtimes. No remote execution needed — it spawns test commands directly.
  - Note: clippy.toml must allow Command::new for this crate (unlike shedul3r which bans it)

- **Standardized JSON output**: Both subcommands output JSON to stdout. The pipeline reads this directly — no file parsing needed.

## Architectural Considerations

- t3str runs on the worker, not in the pipeline orchestrator. The pipeline sends `t3str run --lang rust --repo /tmp/clone/xyz` as the shedul3r command instead of a Python script.
- The pipeline (websmasher) will need updating to call t3str instead of generating Python, but that's a separate task after t3str is built and tested.
- tree-sitter grammars are C libraries compiled via cc crate. The worker Dockerfile needs a C compiler (already has one for other builds).
- Each language module in extract/ and run/ is self-contained. Adding a new language = adding one file to each adapter crate + updating the match in lib.rs.

## Risks & Edge Cases

- **tree-sitter grammar availability**: All 9 languages have well-maintained tree-sitter grammars. Elixir's is less mature but exists.
- **Test framework detection**: Some repos use non-standard test runners. The run adapter needs to detect which framework is present (pytest vs unittest, jest vs mocha vs vitest, phpunit vs nette tester).
- **Isolated environment activation**: Each language module in run/ must activate the isolated env (.venv, .gopath, etc.) before executing tests. This is process env setup, not global state.
- **Large test output**: Some repos produce megabytes of test output. Parsers must stream or cap memory usage.
- **Binary size**: tree-sitter grammars add significant binary size. May need to evaluate whether all 9 are needed in the initial release or if we can start with the 5 most common (Python, Go, Rust, JS, PHP).

## Step-by-step plan

### Phase 1: Scaffold (this task)
1. Create `apps/t3str/` directory structure with all crates
2. Set up workspace Cargo.toml with shared lints, dependencies
3. Copy and adapt deny.toml, clippy.toml, rustfmt.toml, rust-toolchain.toml from shedul3r
4. Write CLAUDE.md with t3str-specific agent instructions
5. Implement domain types: Language, TestFile, TestResult, TestSuite, T3strError
6. Define port traits: TestDiscoverer, TestExecutor
7. Stub out all adapter crates with minimal implementations
8. Wire CLI with clap (extract + run subcommands)
9. Verify: `cargo build`, `cargo clippy`, `cargo test`, `cargo deny check` all pass

### Phase 2: Run adapter (highest value — replaces Python scripts)
1. Implement process execution for each language (spawn test commands, capture output)
2. Implement parsers: JUnit XML, go test JSON, cargo test text, jest JSON, rspec JSON, dotnet stdout, Nette Tester
3. Wire test framework detection per language
4. Add integration tests with fixture repos
5. Verify against the same 21 libraries the current pipeline processes

### Phase 3: Extract adapter (test filtering)
1. Add tree-sitter dependencies and language grammars
2. Implement AST-based test file discovery per language
3. Implement topic-based filtering (find test functions that reference a given symbol/module)
4. Add tests with fixture code snippets
5. Wire extract output into run's --filter flag

### Phase 4: Pipeline integration
1. Update websmasher s06_run_parsers.rs to call `t3str run` instead of generating Python
2. Add `t3str` to worker Dockerfile
3. Run full pipeline, compare results with current Python-based approach
4. Remove Python script generation code

# Extract resilience patterns from shedul3r into limit3r package

**Date:** 2026-03-15 12:25
**Task:** Move rate limiter, circuit breaker, bulkhead, and retry patterns from shedul3r into the shared limit3r package.

## Goal
limit3r becomes a self-contained resilience library with config types, error types, traits, and in-memory implementations. shedul3r re-exports from limit3r so existing code doesn't break.

## Approach

### Step 1: Build limit3r package (9 source files)
- Add serde + thiserror deps to Cargo.toml
- duration_serde.rs — moved from domain/types
- config.rs — moved from domain/types, using crate::duration_serde
- error.rs — NEW Limit3rError enum with 4 variants
- traits.rs — moved from ports/outbound/repo, using Limit3rError + limit3r config types
- rate_limiter.rs, circuit_breaker.rs, bulkhead.rs, retry.rs — moved from adapters/outbound/db, using crate:: imports
- lib.rs — re-exports everything

### Step 2: Update shedul3r to depend on limit3r

- domain-types/src/duration_serde.rs: replace with re-export

### Key decisions
- **Re-export approach:** shedul3r re-exports limit3r types so consumers (commands, api) don't need to change their imports
- **Error mapping:** Add `Resilience(#[from] limit3r::Limit3rError)` variant to SchedulrError for transparent conversion
- **RetryExecutor generic error:** The retry trait needs to be generic over error type since shedul3r's retry uses SchedulrError in the closure, not Limit3rError. Will make RetryExecutor generic: `execute_with_retry<F, Fut, T, E>` where E: From<Limit3rError>

Wait — looking more carefully at the retry trait. The current signature uses `SchedulrError` everywhere. In limit3r, the trait should use `Limit3rError` for the RetryExhausted error, but the closure returns a user-defined error. The cleanest approach: make the error type generic with a bound `E: From<Limit3rError>` so the retry executor can convert its own RetryExhausted into the caller's error type.

Actually, simpler: keep the trait returning a Box<dyn Error> or... No. Let me look again.

The retry executor returns `Result<T, SchedulrError>` and the closure also returns `Result<T, SchedulrError>`. In limit3r, the trait should be generic over the error type. The RetryExhausted gets returned when retries are exhausted, and it needs to be convertible to whatever error the caller uses. So: `E: From<Limit3rError> + Send` constraint on the error type.

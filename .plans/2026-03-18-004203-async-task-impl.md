# Implement async task submission + polling for shedul3r

**Date:** 2026-03-18 00:42
**Task:** Add POST /api/tasks/async and GET /api/tasks/{id} endpoints

## Goal
Clients behind proxies can submit long-running tasks without holding HTTP connections open. Submit returns 202 + task_id, poll returns status.

## Approach

### Architecture decision
- **Domain types** (`async_task.rs`): `AsyncTaskState` enum, `AsyncTaskEntry` struct, `AsyncTaskStatus` response struct -- pure data, serde-serializable
- **REST adapter state** (`state.rs`): `AsyncTaskStore` with `RwLock<BTreeMap>` -- follows `BundleStore` pattern
- **REST handler** (`handlers/async_tasks.rs`): submit + poll endpoints
- **main.rs**: wire store into AppState, spawn reaper, register routes

### Key decisions
- AsyncTaskStore lives in REST adapter (not domain) because polling is a transport concern
- AsyncTaskState/AsyncTaskStatus go in domain-types because they're value objects used by SDK too
- parking_lot::RwLock needed in domain-types for the store... NO: store is in REST adapter which already has parking_lot
- Actually: put the data types (state enum, status struct) in domain-types; put the store in state.rs

### Route conflict avoidance
- GET /api/tasks/status already exists (scheduler status)
- GET /api/tasks/{id} must not match "status" or "limiter-status"
- Use actix-web `.guard()` or register specific routes BEFORE the wildcard
- Actually: register `/api/tasks/async/{id}` to avoid conflicts entirely -- cleaner namespace

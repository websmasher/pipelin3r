## Design Plan: Image Generation Block for pipelin3r

### 1. Understanding the Requirements

The image generation block is a new pipeline step type alongside the existing `agent`, `command`, and `transform` blocks. It makes direct HTTP calls to the OpenRouter API (which proxies to models like Gemini, Flux, DALL-E, GPT-5 Image) to generate images from text prompts with optional reference images. Key differentiators from the agent block:

- Direct HTTP call, not a subprocess via shedul3r
- Binary output (images), not text
- No conversation or tool use -- single request/response
- Needs in-process rate limiting (via limit3r), not shedul3r's queue-based rate limiting

### 2. OpenRouter Image Generation API (from documentation research)

**Endpoint:** `POST https://openrouter.ai/api/v1/chat/completions`

**Auth header:** `Authorization: Bearer <OPENROUTER_API_KEY>`

**Request body:**
```json
{
  "model": "google/gemini-3.1-flash-image-preview",
  "modalities": ["image", "text"],
  "messages": [
    {
      "role": "user",
      "content": [
        { "type": "image_url", "image_url": { "url": "data:image/png;base64,..." } },
        { "type": "text", "text": "Generate a sunset over mountains" }
      ]
    }
  ],
  "image_config": {
    "aspect_ratio": "16:9",
    "image_size": "1K"
  }
}
```

**Response body (relevant fields):**
```json
{
  "id": "gen-...",
  "choices": [{
    "message": {
      "images": [
        { "image_url": { "url": "data:image/png;base64,..." } }
      ]
    }
  }],
  "usage": { "cost": 0.05 }
}
```

**Supported `aspect_ratio` values:** `1:1`, `2:3`, `3:2`, `3:4`, `4:3`, `4:5`, `5:4`, `9:16`, `16:9`, `21:9` (plus extended ratios on Gemini 3.1 Flash only: `1:4`, `4:1`, `1:8`, `8:1`).

**Supported `image_size` values:** `0.5K` (Gemini 3.1 Flash only), `1K` (default), `2K`, `4K`.

### 3. Architectural Decisions

**Decision 1: Module within pipelin3r, not a separate crate.**
- Rationale: It follows the same pattern as `agent`, `command`, and `transform` -- a pipeline step type. It's not large enough to warrant a separate crate, and it needs access to pipelin3r internals (`fs`, `pool`, `PipelineError`, `Auth`).
- Alternative rejected: Separate crate -- would force all internal types to be public and adds unnecessary packaging overhead for what is one module.

**Decision 2: Does NOT use the Executor for SDK/shedul3r interaction.**
- The `Executor` wraps the shedul3r SDK client, which is irrelevant for direct HTTP calls. However, the `Executor` serves as the entry point for creating builders (`executor.agent()`, `executor.command()`). For consistency, image gen should also be created via the executor: `executor.image_gen("step-name")`.
- The Executor provides: default auth, dry-run support. Both are relevant to image gen.

**Decision 3: New reqwest dependency on pipelin3r directly.**
- Currently pipelin3r depends on `shedul3r-rs-sdk` which uses reqwest, but pipelin3r itself doesn't have a direct reqwest dependency. Image gen needs to make HTTP calls directly, so reqwest must be added to pipelin3r's `Cargo.toml`.
- The `reqwest::Client` should be held on the `Executor` (lazy-initialized or built at construction) so it can be shared across image gen calls for connection pooling.

**Decision 4: Rate limiting via limit3r directly (new dependency).**
- pipelin3r does not currently depend on limit3r. Adding `limit3r` as a dependency is natural since they're in the same workspace.
- The `InMemoryRateLimiter` with a `RateLimitConfig` will be used. The rate limiter instance lives on the `Executor` so it persists across calls.
- Default config: something conservative like 10 requests per 60 seconds, configurable via the builder.

**Decision 5: Auth reuses pipelin3r's Auth system with a new variant.**
- The existing `Auth` enum has `OAuthToken`, `ApiKey`, `FromEnv`, `Custom`. For image gen, the relevant credential is `OPENROUTER_API_KEY`. Two options:
  - (a) Use `Auth::Custom` with a map containing `OPENROUTER_API_KEY` -- works but awkward.
  - (b) Add a dedicated `Auth::OpenRouterKey(String)` variant -- cleaner but modifies the shared Auth enum.
  - (c) Keep auth as a plain `String` API key on the image gen builder, separate from the LLM auth system -- simplest, since OpenRouter auth is just a bearer token, not env vars for a subprocess.
- **Chosen: (c).** The `Auth` enum is designed for subprocess environment injection (the `to_env()` method). Image gen is an HTTP call, not a subprocess. The API key is passed as a header, not an env var. A separate `api_key(&str)` method on the builder is cleaner and avoids polluting the Auth system. The builder can also have a `from_env()` convenience that reads `OPENROUTER_API_KEY`.

**Decision 6: Batch support via the existing `run_pool` function.**
- The steady-parent pipeline generates multiple images sequentially. An `ImageGenBatchBuilder` (mirroring `AgentBatchBuilder`) would use `run_pool` for bounded concurrency, exactly like the agent batch system.

/// Parsed response from OpenRouter image generation.
pub(crate) struct GeneratedImage {

    /// API cost in USD.

The client builds the multipart message content array with image_url parts for reference images and a text part for the prompt, sends the POST, and parses the response to extract the base64-encoded image.

### 6. Error Handling

A new error variant is added to `PipelineError`:

```rust
/// Image generation failed.
#[error("image generation failed: {message}")]
ImageGenFailed {
    /// Description of the failure.
    message: String,
},
```

HTTP errors, rate limit errors (from limit3r), JSON parsing failures, and missing image data in responses all map to this variant.

### 7. Rate Limiting Flow

```
ImageGenBuilder::execute()
    |
    v
rate_limiter.acquire_permission("openrouter", &config).await?
    |
    v
client::generate_image(http_client, api_key, model, prompt, refs, ...)
    |
    v
write output to work_dir
```

If the rate limiter blocks, the call waits (up to timeout) then proceeds. If it times out, a `PipelineError::ImageGenFailed` is returned with a rate limit message.

The `limit3r::Limit3rError::RateLimitExceeded` maps to `PipelineError::ImageGenFailed`.

### 8. Dry-Run Support

Like the agent builder, when the executor is in dry-run mode, `ImageGenBuilder::execute()` captures the request parameters (model, prompt, reference image count, aspect ratio) to a JSON file in the dry-run capture directory instead of making the HTTP call. This mirrors the agent dry-run capture in `agent/execute.rs`.

### 9. work_dir Integration

- Reference images can be loaded from paths relative to or absolute from work_dir
- Output images are written to work_dir with the configured filename
- If work_dir doesn't exist, it's created (using `crate::fs::create_dir_all`)
- The `reference_images_from_dir` method uses `crate::fs::read_dir` and `crate::fs::read` to load image files

### 10. New Dependencies

In `packages/pipelin3r/Cargo.toml`:
```toml
[dependencies]
# ... existing deps ...

serde = { version = "1", features = ["derive"] }

```

- `reqwest` -- HTTP client for OpenRouter API
- `serde` -- serialize/deserialize request/response JSON (pipelin3r already uses `serde_json` but needs `serde` with `derive` for struct annotations)
- `limit3r` -- in-process rate limiting
- `base64` -- decode base64-encoded image data from API response (the `base64` crate is the standard for this; alternatives like using `data_encoding` exist but `base64` is more widely used)

### 12. Risks and Edge Cases

- **Base64 response parsing:** OpenRouter returns images as `data:image/png;base64,...` URLs. The regex extraction pattern from the steady-parent code is robust but should handle edge cases (missing prefix, unexpected MIME types). Using a proper parser rather than regex is safer.
- **Large response bodies:** Generated images can be several MB in base64. reqwest handles this fine but timeout configuration on the HTTP client matters. A 60-second default timeout is reasonable.
- **Model availability:** Models come and go on OpenRouter. The `ImageModel::Custom(String)` variant ensures forward compatibility.
- **Cost tracking:** The `usage.cost` field may be absent or zero. The steady-parent code has a fallback to query `/api/v1/generation?id=...` for cost. This is worth including as a best-effort cost lookup.
- **Image format conversion:** The steady-parent code uses `sharp` to convert to WebP. In Rust, this would require the `image` crate. For V1, we should write the raw bytes as received (PNG from most models) and let the caller convert if needed. A future enhancement could add optional WebP conversion via the `image` crate.

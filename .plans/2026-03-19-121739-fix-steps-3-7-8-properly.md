## Step 4: Clone + Install + Build Wrapper Scripts

For each library in libraries.json:
1. `git clone --depth 1` (already done)
2. Install the parser as a runnable command
3. Generate a wrapper script that takes security.txt on stdin, outputs parsed JSON on stdout

### Per-language install + wrapper strategy

**Python**: `pip install -e <clone_dir>` or `pip install <clone_dir>`. Wrapper: python script that imports the library and calls its parse function.

**Go**: `go build` in the clone dir. Wrapper: compile a small Go main that reads stdin and calls the parser.

**Rust**: `cargo build --release`. Wrapper: compile small binary or use the lib's example/CLI if it has one.

**PHP**: `composer install` in clone dir. Wrapper: php script that requires autoload and calls parser.

**Ruby**: `gem build && gem install`. Wrapper: ruby script that requires the gem and calls parser.

**JavaScript**: `npm install` in clone dir. Wrapper: node script that requires the package and calls parser.

### Wrapper script interface
Every wrapper has the same interface:
- Input: security.txt content on stdin
- Output: JSON on stdout with parsed fields
- Exit code: 0 on success, non-zero on parse error
- Stderr: error messages (ignored by step 8)

The output JSON schema should be normalized:
```json
{
  "fields": [
    {"name": "Contact", "value": "mailto:security@example.com"},
    {"name": "Expires", "value": "2025-01-01T00:00:00Z"}
  ],
  "comments": ["# This is a comment"],
  "errors": []
}
```

### Challenge: each library has a different API
This is the hard part. Each library has its own parse function, its own output types, its own error handling. The wrapper script needs to be library-specific.

**Solution: LLM generates the wrapper script.** This is a perfect use case for a VerifiedStep:
- Doer: Given the library's source code (from step 6) and its API, generate a wrapper script
- Breaker: Run the wrapper on a trivial input (`Contact: mailto:test@example.com\n`) and check it produces valid JSON
- Fixer: Fix the wrapper based on the error

Actually — this adds significant complexity. For MVP, we can use a simpler approach:

**Simpler approach for MVP**: Use the library's existing test runner or CLI if available. Many of these libraries have test suites that demonstrate usage. We can:
1. For libraries with a CLI: use it directly
2. For libraries without: generate a minimal wrapper script per language (LLM step)
3. Validate each wrapper works on a trivial input before proceeding

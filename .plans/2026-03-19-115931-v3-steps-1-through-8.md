**Date:** 2026-03-19 11:59
**Task:** Migrate V3 to a cleanly architected multi-module pipeline with steps 1-8

## Goal
Replace the single-file V3 main.rs with a well-structured Rust binary that runs steps 1-8 of the parser development pipeline. Each step is its own module. Shared config, language definitions, and helpers are factored out.

### Key design decisions

1. **No step trait.** Steps are just async functions `(config) -> Result<(), String>`. A trait adds ceremony without value here — steps have different signatures (some need concurrency, some don't).

2. **Shared `PipelineConfig`** carries: package name, package_dir, executor (Arc), concurrency, prompts_dir. Passed to every step.

3. **Steps 4-5 share a `clone_and_extract` helper** parameterized by mode (tests/source) and output suffix.

4. **Step 6 gets an LLM breaker** — a second agent that reviews the spec reference against the actual RFC. This is the first step to use `Breaker::Agent`.

5. **Steps 7-8 are pure Rust** — no external tools needed for classification. Step 7 needs reference parser wrappers (which step 3 would install, but for now we'll use the existing extract-tools outputs as proxy or skip if parsers aren't installed).

### Step 3 (clone+install) — key change from V2
V2's step 3 was "extract libraries JSON from overview." That's now our step 3.
The NEW pipeline's step 3 is "clone repos from libraries.json + install reference parsers." This is what V2 did in steps 4-5 (the clone_and_extract function), but with an important addition: building wrapper scripts that normalize parser output to stdin→stdout JSON.

For MVP: clone + run extract-tools (same as V2 steps 4-5). The wrapper script building is a future step.

### Steps 7-8 — what we can do now
Without installed reference parsers with wrapper scripts, we can't fully run step 7. But we can:
- Step 7: Use the extracted test data from step 4 as a proxy (the test assertions tell us what the parser expects)
- Step 8: Classify based on agreement across libraries' test expectations

Actually, for an honest MVP: steps 7-8 require running actual parsers. We should implement the infrastructure (directory structure, classification logic) but acknowledge that without wrapper scripts, the ground truth is from test assertions, not parser execution.

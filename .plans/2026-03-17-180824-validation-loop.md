# Validation + Remediation Loop for pipelin3r

## Design Plan

### Goal

Add a `ValidateLoop` composition block that automates the validate-fix-revalidate cycle. It takes a validator function (user-provided, returns structured findings), an agent configuration for fixes, and loops until validation passes or max iterations are reached. This eliminates ad-hoc reimplementation of this pattern across consuming projects like dev-process and steady-parent.

### Input Information

**Existing codebase patterns observed:**

1. **Builder pattern everywhere.** `AgentBuilder`, `AgentBatchBuilder`, `CommandBuilder`, `TransformBuilder` all follow the same shape: struct with setter methods, terminal `.execute()` that returns a typed result.

2. **`AgentResult` is the universal agent output** -- `success: bool` + `output: String`. The fix agent will produce this.

3. **`CommandResult` exists for shell commands** -- used by the validate step when validation means "run cargo test."

4. **`Executor` owns the SDK client** and is borrowed by all builders. The validation loop must borrow the executor too.

5. **`PipelineError` is the error type** everywhere, using `thiserror`. New error variants go here.

6. **500-line file limit.** The agent module is already at 482 lines in `mod.rs` and 486 in `execute.rs`. The validation loop MUST be a separate module, not added to existing files.

7. **No `HashMap`, no `unwrap`, no `expect`, no `unsafe`, no `anyhow`.** All arithmetic must use checked/saturating ops.

8. **`work_dir` is the standard context-passing mechanism** -- a directory path that gets passed to agents and commands.

9. **Real-world usage** (from dev-process CLAUDE.md): step 18 implements parser, step 19 runs `cargo test`, failures would feed back to a fix agent. This is sequential, not batched.

### Design Decisions

**Q1: What is the validation report type?**

A `ValidationReport` struct containing a `Vec<ValidationFinding>`. Each finding has an `id` (for tracking across iterations), `severity` (error/warning), `message`, and optional `context` (file path, line number, etc.). This is structured enough to format into prompts but generic enough for both "cargo test failures" and "caption quality checks."

The validator is a user-supplied async closure that returns `Result<ValidationReport, PipelineError>`. Not a trait -- closures compose better with the builder pattern and match how `TransformBuilder::apply` and `AgentBatchBuilder::for_each` work.

**Q2: How does the fix prompt get constructed from the validation report?**

The user provides a "prompt builder" closure: `Fn(&ValidationReport, iteration) -> String`. This gives full control over how findings are formatted into the fix agent's prompt. The library provides a default `ValidationReport::to_markdown()` method that renders findings in a standard format, so simple cases can just do `|report, _| format!("Fix these:\n{}", report.to_markdown())`.

**Q3: Does the fix agent work in the same work_dir?**

Yes. The fix agent operates in the same `work_dir` as the original output. The whole point is that the agent modifies files in-place and then re-validation checks the same directory. This matches both real-world patterns (dev-process: cargo test runs in the package dir, fix agent edits source in the same dir).

**Q4: What happens when max iterations are exhausted?**

Return `Ok(ValidateLoopResult)` with `converged: false` and the last `ValidationReport`. NOT an error -- the caller decides what to do with partial results. Some callers may want to continue with warnings; others may want to abort. Making this an error forces the caller to handle it in a catch-all error handler, which is the wrong place for this decision.

**Q5: Single function/builder or trait-based?**

Builder. Consistent with every other composition block in the crate. `ValidateLoopBuilder` follows the same pattern as `AgentBuilder`.

### API Sketch

```rust
// ── packages/pipelin3r/src/validate_loop/mod.rs ──

/// A single finding from validation.
#[derive(Debug, Clone)]
pub struct ValidationFinding {
    /// Stable identifier for tracking this finding across iterations.
    pub id: String,
    /// Severity level.
    pub severity: FindingSeverity,
    /// Human-readable description of the problem.
    pub message: String,
    /// Optional structured context (file path, line number, etc.).
    pub context: Option<FindingContext>,
}

/// Severity of a validation finding.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum FindingSeverity {
    /// Must be fixed before validation can pass.
    Error,
    /// Informational; does not block validation from passing.
    Warning,
}

/// Optional location/context for a finding.
#[derive(Debug, Clone)]
pub struct FindingContext {
    /// File path (relative to work_dir).
    pub file: Option<String>,
    /// Line number (1-based).
    pub line: Option<u32>,
    /// Raw output snippet (e.g., compiler error text).
    pub snippet: Option<String>,
}

/// Outcome of a validation pass.
#[derive(Debug, Clone)]
pub struct ValidationReport {
    /// All findings from this validation pass.
    pub findings: Vec<ValidationFinding>,
}

impl ValidationReport {
    /// Whether validation passed (no error-severity findings).
    pub fn passed(&self) -> bool { ... }

    /// Count of error-severity findings.
    pub fn error_count(&self) -> usize { ... }

    /// Count of warning-severity findings.  
    pub fn warning_count(&self) -> usize { ... }

    /// Render findings as markdown for inclusion in an agent prompt.
    pub fn to_markdown(&self) -> String { ... }
}

/// Result of the validate-fix loop.
#[derive(Debug, Clone)]
pub struct ValidateLoopResult {
    /// Whether validation passed within the iteration limit.
    pub converged: bool,
    /// Number of iterations executed (1 = validated once, no fixes needed).
    pub iterations: u32,
    /// The final validation report.
    pub final_report: ValidationReport,
    /// Reports from each iteration, for observability.
    pub iteration_reports: Vec<ValidationReport>,
}

/// Builder for the validate-fix loop.
#[must_use]
pub struct ValidateLoopBuilder<'a> {
    executor: &'a Executor,
    name: String,
    work_dir: Option<PathBuf>,
    max_iterations: u32,  // default: 3
    // Validator closure: given a work_dir, produce a report.
    validator: Option<Box<dyn Fn(&Path) -> BoxFuture<'_, Result<ValidationReport, PipelineError>> + Send + Sync>>,
    // Prompt builder: given the report and iteration number, produce the fix prompt.
    prompt_builder: Option<Box<dyn Fn(&ValidationReport, u32) -> String + Send + Sync>>,
    // Agent configuration for the fix step (inherited from AgentBuilder pattern).
    auth: Option<&'a Auth>,
    model: Option<Model>,
    timeout: Option<Duration>,
    tools: Option<String>,
    expected_outputs: Vec<String>,
}

impl<'a> ValidateLoopBuilder<'a> {
    pub(crate) fn new(executor: &'a Executor, name: &str) -> Self { ... }

    /// Set the work directory for validation and fixing.
    pub fn work_dir(mut self, path: &Path) -> Self { ... }

    /// Set the maximum number of fix iterations (default: 3).
    pub fn max_iterations(mut self, n: u32) -> Self { ... }

    /// Set the validator function.
    ///
    /// The validator receives the work directory path and returns a
    /// `ValidationReport`. If the report has no error-severity findings,
    /// the loop terminates successfully.
    pub fn validator<F, Fut>(mut self, f: F) -> Self
    where
        F: Fn(&Path) -> Fut + Send + Sync + 'static,
        Fut: Future<Output = Result<ValidationReport, PipelineError>> + Send + 'static,
    { ... }

    /// Set the prompt builder that formats validation failures into
    /// the fix agent's prompt.
    ///
    /// Receives the current `ValidationReport` and the iteration number
    /// (0-indexed). Must return the complete prompt string.
    pub fn prompt_builder<F>(mut self, f: F) -> Self
    where
        F: Fn(&ValidationReport, u32) -> String + Send + Sync + 'static,
    { ... }

    /// Set the LLM model for the fix agent.
    pub fn model(mut self, model: Model) -> Self { ... }

    /// Set the timeout for the fix agent.
    pub const fn timeout(mut self, timeout: Duration) -> Self { ... }

    /// Set the allowed tools for the fix agent.
    pub fn tools(mut self, tools: &[Tool]) -> Self { ... }

    /// Override authentication for the fix agent.
    pub const fn auth(mut self, auth: &'a Auth) -> Self { ... }

    /// Set expected output files for the fix agent (remote transport hint).
    pub fn expect_outputs(mut self, paths: &[&str]) -> Self { ... }

    /// Execute the validate-fix loop.
    ///
    /// 1. Run validator
    /// 2. If passed → return success
    /// 3. If failed → build fix prompt from report → run fix agent
    /// 4. Re-validate → repeat until pass or max_iterations exhausted
    pub async fn execute(self) -> Result<ValidateLoopResult, PipelineError> {
        let work_dir = self.work_dir.ok_or_else(|| ...)?;
        let validator = self.validator.ok_or_else(|| ...)?;
        let prompt_builder = self.prompt_builder.ok_or_else(|| ...)?;
        
        let mut iteration_reports = Vec::new();
        let mut current_iteration: u32 = 0;

        loop {
            // Run validation
            let report = validator(&work_dir).await?;
            iteration_reports.push(report.clone());
            
            if report.passed() {
                return Ok(ValidateLoopResult {
                    converged: true,
                    iterations: current_iteration.saturating_add(1),
                    final_report: report,
                    iteration_reports,
                });
            }

            // Check iteration limit
            if current_iteration >= self.max_iterations {
                return Ok(ValidateLoopResult {
                    converged: false,
                    iterations: current_iteration.saturating_add(1),
                    final_report: report,
                    iteration_reports,
                });
            }

            // Build fix prompt and run fix agent
            let fix_prompt = prompt_builder(&report, current_iteration);
            
            tracing::info!(
                "[validate-loop] '{}' iteration {}: {} errors, running fix agent",
                self.name, current_iteration, report.error_count()
            );

            let agent_result = self.executor
                .agent(&format!("{}-fix-{}", self.name, current_iteration))
                .model(...)  // from self
                .timeout(...)
                .tools(...)
                .auth(...)
                .work_dir(&work_dir)
                .expect_outputs(...)
                .prompt(&fix_prompt)
                .execute()
                .await?;

            if !agent_result.success {
                tracing::warn!(
                    "[validate-loop] '{}' fix agent failed on iteration {}",
                    self.name, current_iteration
                );
            }

            current_iteration = current_iteration.saturating_add(1);
        }
    }
}
```

   ```rust

   #[error("validation loop '{name}' did not converge after {iterations} iterations: {remaining_errors} errors remain")]
   ValidationExhausted {
       name: String,
       iterations: u32,
       remaining_errors: usize,
   },
   ```
   Note: this variant is optional -- the builder returns `Ok(ValidateLoopResult { converged: false })` instead of erroring, but having the variant available lets callers convert to an error if they want: `result.require_converged()?`.

4. **`Cargo.toml`** -- No new dependencies needed. The `futures` crate or a `BoxFuture` type is needed for the validator closure. Two options:
   - Add `futures-core = "0.3"` for `BoxFuture` (minimal dep, widely used)
   - Use `Pin<Box<dyn Future>>` directly to avoid the dependency

   Recommendation: use `Pin<Box<dyn Future<Output = ...> + Send + '_>>` directly. One type alias in the module keeps it clean, and avoids adding a dependency to a published crate.

### Architectural Considerations

**Why a builder and not a free function?** The fix agent needs model, timeout, tools, auth, expected_outputs configuration. A free function would need 10+ parameters. The builder pattern matches every other composition block.

**Why closures instead of traits?** Traits would require the user to define a struct and impl block for every validator. Closures are more ergonomic for the common case and match `TransformBuilder::apply` and `AgentBatchBuilder::for_each`. Power users can still wrap a trait object in a closure.

**Why async validator?** The primary use case (run `cargo test`) is inherently async. Making it sync would force `block_on` inside the loop.

**Why `converged: bool` instead of `Result`?** Exhausting iterations is not exceptional -- it is an expected outcome the caller must handle. Making it an error conflates "the loop machinery broke" with "the code has bugs the agent could not fix." The `require_converged()` convenience method bridges the gap for callers who want the error path.

**Dry-run support:** The builder delegates to `executor.agent()` which already handles dry-run. The validator closure runs normally in dry-run mode (it is user code), but the fix agent invocation will capture to disk instead of executing. This is correct behavior -- in dry-run mode you want to see what the fix prompts would look like.

### Risks and Edge Cases

1. **Validator closure lifetime:** The validator borrows `&Path` which is owned by the builder. The closure must return a `Send + 'static` future but receive a borrowed path. Solution: clone the path into the future inside the closure wrapper.

2. **Fix agent does not actually fix anything:** The loop will run max_iterations times with the same failures. The `iteration_reports` vec lets callers detect this (compare error counts across iterations).

3. **Validation itself fails (I/O error, etc.):** The `?` propagates immediately -- this is a hard error, not a validation failure.

4. **Zero max_iterations:** Should mean "validate once, never fix." The loop runs the validator once and returns immediately. Document this.

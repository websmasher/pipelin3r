# t3str Phase 2 — Run adapter implementation

**Date:** 2026-03-20 09:56
**Task:** Implement all 9 language executors and output parsers in the t3str run adapter, replacing the Python-in-Rust-format-macro approach.

## Goal

Every language module in `apps/t3str/crates/adapters/outbound/run/src/` executes tests and parses structured output. The parsers use proper XML/JSON libraries (quick-xml, serde_json), not regex. The output is a unified `TestSuite` JSON struct.

## Input Information

Full specification extracted from the current Python implementation in websmasher's s06_run_parsers.rs and config.rs. See research agent output for per-language details.

Key edge cases already solved in the Python implementation that must be preserved:
- Pytest exit code 5 fallback (explicit test file discovery)
- Go JSON case sensitivity + deduplication by (Package, Test)
- Rust cargo config backup/restore
- PHP Nette Tester vs PHPUnit detection
- C# TRX one-per-project bug → stdout fallback
- JS npm test first, then Jest JSON
- Lock file removal for all languages
- Zero-test validation (broken library vs pipeline bug)

## Approach

### Shared parsers (parsers/ module)

These are the reusable output parsers. No regex — proper structured parsing.

| Parser | Format | Used by | Library |
|--------|--------|---------|---------|
| `junit_xml.rs` | JUnit XML | Python, PHP, Java | quick-xml + serde |
| `go_json.rs` | `go test -json` NDJSON | Go | serde_json line-by-line |
| `cargo_text.rs` | `cargo test` text | Rust | string matching (structured patterns) |
| `dotnet_stdout.rs` | `Passed/Failed` lines | C# | string matching |
| `rspec_json.rs` | RSpec JSON | Ruby | serde_json |
| `jest_json.rs` | Jest JSON | JavaScript | serde_json |
| `nette_text.rs` | Nette Tester console | PHP | string matching |
| `mocha_text.rs` | Mocha checkmark output | JavaScript | string matching (unicode checkmarks) |
| `mix_text.rs` | `mix test` output | Elixir | string matching |

String matching parsers use structured line-by-line parsing with specific patterns — NOT regex. They match on known prefixes/suffixes (`"test "`, `" ... ok"`, `"  Passed "`, etc.).

### Per-language executors

Each language module:
1. Sets up isolated environment (env vars, paths)
2. Detects test framework (which runner to use)
3. Spawns the test command via `tokio::process::Command`
4. Captures stdout + stderr
5. Reads any output files (JUnit XML, JSON, TRX)
6. Calls the appropriate parser
7. Falls back to secondary parser if primary returns empty
8. Returns `TestSuite`

### Execution order

Dependencies flow: parsers first (shared), then language modules (use parsers).

1. **Batch 1 — Parsers** (all independent, parallel):
   - junit_xml.rs, go_json.rs, cargo_text.rs, dotnet_stdout.rs, rspec_json.rs, jest_json.rs, nette_text.rs, mocha_text.rs, mix_text.rs

2. **Batch 2 — Language modules** (all independent, parallel):
   - python.rs, go.rs, rust_lang.rs, php.rs, csharp.rs, javascript.rs, ruby.rs, java.rs, elixir.rs

3. **Batch 3 — Integration + Cargo.toml updates**:
   - Update run crate Cargo.toml with quick-xml, serde_json deps
   - Update lib.rs to wire everything
   - Verify full workspace compiles

### Key decisions

- **No regex crate.** Banned in deny.toml. String matching parsers use `str::starts_with`, `str::contains`, `str::split`, `str::trim` — standard library string methods.
- **quick-xml for JUnit/TRX.** Proper XML parsing, not string matching on angle brackets.
- **serde_json for Go/Jest/RSpec JSON.** Proper deserialization into typed structs.
- **tokio::process::Command for execution.** Async process spawning, captures stdout+stderr separately.
- **Parsers are pure functions.** `fn parse(input: &str) -> Result<Vec<TestResult>, T3strError>`. No side effects, easy to test with fixtures.
- **File reading in language modules, not parsers.** Language module reads the output file, passes content to parser. Parsers only work with strings.

## Risks & Edge Cases

- **Binary output.** Some test runners produce non-UTF8 output. Use `String::from_utf8_lossy`.
- **Large output.** Cap raw_output at last 2000 chars (matching current behavior).
- **Timeout.** Each language module should have a configurable timeout (default 300s, 600s for Rust/C#).
- **Missing runtimes.** If a language runtime isn't installed, return `T3strError::ExecutionFailed` with clear message.

## Architectural Considerations

- Parsers are in a `pub mod parsers` so they can be tested independently and potentially reused by extract adapter
- Each parser function takes `&str` and returns `Result<Vec<TestResult>>` — pure transformation
- Language modules own the process lifecycle and file I/O — adapter concerns stay in adapters
- No domain or port types need modification

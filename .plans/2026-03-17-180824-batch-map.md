## Design: Batch-Map Composition Block for pipelin3r

### Analysis of What Exists

The existing `AgentBatchBuilder` already provides:
- Take a `Vec<T>`, apply a `for_each` closure mapping each `T` to an `AgentTask` (with prompt, work_dir, expected_outputs, auth override)
- Bounded concurrency via `run_pool`
- Per-item `Result<AgentResult, PipelineError>` collection
- Duplicate work_dir detection
- Dry-run capture mode
- Partial failure reporting (log warning, but return all results)

What it does NOT provide:
1. **Item discovery** -- callers must build the `Vec<T>` themselves (glob directories, parse JSON manifests, etc.)
2. **Chunking/splitting** -- the dev-process has its own `split_into_agent_tasks` for splitting large payloads into agent-sized chunks; this pattern is not in pipelin3r
3. **Progress reporting** -- no callback or channel for "N of M done"
4. **Generic batch beyond agents** -- there is no `CommandBatchBuilder` or `TransformBatchBuilder`; only agents support batch mode
5. **Structured per-item results with item identity** -- results are `Vec<Result<AgentResult, PipelineError>>` keyed by index, but the caller has no easy way to correlate results back to items since the mapper consumes the item

### Key Design Questions Answered

**Q1: Does `AgentBatchBuilder` cover this already?**
Mostly yes for the agent case. The core map-and-execute loop is solid. What is missing is: (a) discovery helpers, (b) chunking, (c) progress reporting, (d) result-item correlation.

**Q2: Should batch-map be agent-specific or generic?**
It should be generic. The real-world patterns show that the same fan-out pattern applies to commands (e.g., "run cargo check on N packages") and transforms (e.g., "process N JSON files"). However, the existing `AgentBatchBuilder` is tightly coupled to agent execution (task YAML, SDK client, auth, bundle transport). The right approach is a **generic `BatchMap` composition block** that works with any async function, and then the existing `AgentBatchBuilder` can be reimplemented on top of it (or kept as a convenience layer that delegates to `BatchMap` internally).

**Q3: How does item discovery work?**
It should be a separate concern -- utility functions that produce `Vec<T>` that can be fed into `BatchMap`. This keeps the batch-map block focused on execution and lets discovery be composable.

**Q4: Should there be a chunking helper?**
Yes, as a standalone utility. The dev-process `split_into_agent_tasks` is too domain-specific (test files, .rs extensions), but the core algorithm (split a list into size-bounded chunks) is reusable.

### Proposed API

```rust
// ── packages/pipelin3r/src/batch/mod.rs ──

/// Result of a batch-map execution, preserving item identity.
pub struct BatchMapResult<T, R> {
    /// Per-item outcomes, paired with the original item for correlation.
    pub results: Vec<BatchItemOutcome<T, R>>,
}

/// One item's outcome in a batch-map execution.
pub struct BatchItemOutcome<T, R> {
    /// The original item (or a key derived from it).
    pub item: T,
    /// The execution result for this item.
    pub result: Result<R, PipelineError>,
}

impl<T, R> BatchMapResult<T, R> {
    /// Count succeeded vs failed.
    pub fn counts(&self) -> (usize, usize) { ... }
    
    /// Iterate over successful results only.
    pub fn successes(&self) -> impl Iterator<Item = (&T, &R)> { ... }
    
    /// Iterate over failures only.
    pub fn failures(&self) -> impl Iterator<Item = (&T, &PipelineError)> { ... }
    
    /// Require all items succeeded, or return BatchPartialFailure.
    pub fn require_all_succeeded(&self) -> Result<(), PipelineError> { ... }
}

/// Builder for batch-map execution over a collection of items.
pub struct BatchMap<T, K, R> {
    items: Vec<T>,
    concurrency: usize,
    key_fn: Option<Box<dyn Fn(&T) -> K + Send + Sync>>,
    progress_fn: Option<Box<dyn Fn(BatchProgress) + Send + Sync>>,
}

/// Progress update emitted during batch execution.
pub struct BatchProgress {
    pub completed: usize,
    pub failed: usize,
    pub total: usize,
}

impl<T: Send + 'static> BatchMap<T, usize, ()> {
    /// Create a new batch-map over the given items.
    pub fn new(items: Vec<T>, concurrency: usize) -> Self { ... }
}

impl<T, K, R> BatchMap<T, K, R> 
where
    T: Send + 'static,
    K: Send + 'static,
    R: Send + 'static,
{
    /// Set an optional progress callback.
    pub fn on_progress<F>(mut self, f: F) -> Self
    where F: Fn(BatchProgress) + Send + Sync + 'static { ... }

    /// Execute an async function for each item with bounded concurrency.
    /// Returns a BatchMapResult pairing each item with its outcome.
    pub async fn execute<F, Fut, Res>(self, f: F) -> BatchMapResult<T, Res>
    where
        F: Fn(T) -> Fut + Send + Sync + Clone + 'static,
        Fut: Future<Output = Result<Res, PipelineError>> + Send,
        Res: Send + 'static,
    { ... }
}
```

Usage example -- agent batch (how `AgentBatchBuilder` could delegate):

```rust
let results = BatchMap::new(languages, 4)
    .on_progress(|p| tracing::info!("{}/{} done", p.completed, p.total))
    .execute(|lang| {
        let exec = executor.clone(); // or Arc
        async move {
            exec.agent("find-libraries")
                .prompt(&format!("Find libraries for {lang}"))
                .work_dir(&prepare_work_dir(&lang)?)
                .execute()
                .await
        }
    })
    .await;
```

Usage example -- command batch:

```rust
let results = BatchMap::new(packages, 4)
    .execute(|pkg| async move {
        let cmd = CommandBuilder::new("cargo")
            .args(&["check", "--manifest-path", &pkg.manifest])
            .execute()
            .await?;
        cmd.require_success()?;
        Ok(cmd)
    })
    .await;
```

### Relationship to Existing Code

1. **`run_pool`** stays as-is. `BatchMap::execute` delegates to `run_pool` internally, wrapping it with progress reporting and result-item pairing.

2. **`AgentBatchBuilder`** stays as-is for now. It already works well for the agent-specific case. In a future iteration it could be reimplemented on top of `BatchMap`, but that is a refactor, not a new feature. The two can coexist: `AgentBatchBuilder` is the high-level agent-specific API; `BatchMap` is the low-level generic API for when you need command batches, transform batches, or mixed batches.

3. **New public exports** from `lib.rs`:
   - `pub mod batch;`
   - Re-export `BatchMap`, `BatchMapResult`, `BatchItemOutcome`, `BatchProgress`
   - Re-export discovery and chunking functions

### Dependencies

- `glob` crate for `from_glob` discovery (check if already in Cargo.toml, or add it)
- `serde` + `serde_json` for `from_json` discovery (already present)
- No new async runtime dependencies; uses existing `tokio` + `run_pool`

### Risks and Edge Cases

1. **Item ownership**: `BatchMap::execute` consumes items. If the caller needs items back for correlation, they need to clone before or use `BatchItemOutcome`. The design returns items back in `BatchItemOutcome` so no pre-cloning is needed.

2. **Progress callback thread safety**: The progress callback fires from the pool's async tasks. It must be `Send + Sync`. Using `Arc<AtomicUsize>` counters internally, calling the callback after each task completes.

3. **Glob dependency**: Adding a `glob` crate dependency. Should check `deny.toml` compatibility. Alternative: use `std::fs::read_dir` with manual pattern matching to avoid the dependency, but glob patterns are significantly more ergonomic.

4. **500-line file limit**: The batch module is split into three files, each well under 500 lines.

5. **No `unwrap`/`expect`**: All fallible operations use `?` or `.ok_or_else()`.

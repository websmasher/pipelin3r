# Feature: Jitter Support for All Timed Components

## Goal

Add configurable jitter to all limit3r components that use timed delays: RetryExecutor, RateLimiter, and CircuitBreaker. Deterministic delays are a fingerprint in anti-bot scenarios and cause thundering-herd problems in distributed systems. Jitter randomizes timing to solve both.

## Motivation

All current delays in limit3r are deterministic:
- `RetryExecutor::compute_delay()` → exact `wait_duration * multiplier^attempt`
- `RateLimiter::acquire_permission()` → exact `sleep_until(next_window)`
- `CircuitBreaker` → exact `wait_duration_in_open_state`

This is a problem for two use cases:
1. **Anti-bot evasion** — Cloudflare and similar systems fingerprint request timing patterns. Deterministic exponential backoff is a known bot signature.
2. **Distributed retry storms** — When N clients fail simultaneously and retry with identical backoff, they all retry at the same instant, causing another failure wave.

## Approach

### Jitter type

**Full jitter** (also called "equal jitter"): given a computed delay `d` and a jitter factor `j` (0.0 to 1.0), the actual delay is uniformly random in `[d * (1 - j), d * (1 + j)]`.

Example: delay = 2s, jitter_factor = 0.5 → actual delay is random in [1s, 3s].

Why full jitter over decorrelated jitter: simpler, more predictable bounds, sufficient for both use cases. AWS recommends full jitter for most retry scenarios.

### Config changes

Add `jitter_factor: f64` to three config structs:

```rust
// RetryConfig — jitter on backoff delay between retries
pub struct RetryConfig {
    pub max_attempts: u32,
    pub wait_duration: Duration,
    pub backoff_multiplier: f64,
    pub max_delay: Duration,
    pub jitter_factor: f64,  // NEW: 0.0 = no jitter (default), 0.5 = ±50%
}

// RateLimitConfig — jitter on wait-for-window sleep
pub struct RateLimitConfig {
    pub limit_for_period: u32,
    pub limit_refresh_period: Duration,
    pub timeout_duration: Duration,
    pub jitter_factor: f64,  // NEW: 0.0 = no jitter (default)
}

// CircuitBreakerConfig — jitter on open→half-open transition wait
pub struct CircuitBreakerConfig {
    pub failure_rate_threshold: f64,
    pub sliding_window_size: u32,
    pub wait_duration_in_open_state: Duration,
    pub jitter_factor: f64,  // NEW: 0.0 = no jitter (default)
}
```

**BulkheadConfig does NOT get jitter.** Bulkhead is a semaphore — its `max_wait_duration` is a timeout, not a delay. Adding jitter to a timeout makes no sense.

### Default value

`jitter_factor` defaults to `0.0` (no jitter). This preserves backward compatibility — existing behavior is identical. Callers opt into jitter by setting a nonzero value.

Serde: use `#[serde(default)]` so existing serialized configs without `jitter_factor` deserialize correctly as 0.0.

### Validation

Add to each config's `validate()`:
```rust
if self.jitter_factor < 0.0 || self.jitter_factor > 1.0 {
    return Err("jitter_factor must be between 0.0 and 1.0 inclusive".to_owned());
}
```

### Random number generation

Use `rand` crate with `thread_rng()` for jitter randomization. This is cryptographically unnecessary — `SmallRng` or `ThreadRng` is fine for timing jitter.

Add to `Cargo.toml`:
```toml
rand = "0.9"
```

### Implementation: shared jitter function

Create a utility function used by all components:

**File:** `src/jitter.rs` — NEW

```rust
use std::time::Duration;
use rand::Rng;

/// Apply jitter to a duration.
///
/// Given a base duration and a jitter factor (0.0 to 1.0), returns a
/// random duration uniformly distributed in `[base * (1 - factor), base * (1 + factor)]`.
///
/// A jitter_factor of 0.0 returns the base duration unchanged.
/// A jitter_factor of 0.5 returns a duration in [base * 0.5, base * 1.5].
/// A jitter_factor of 1.0 returns a duration in [0, base * 2.0].
pub(crate) fn apply_jitter(base: Duration, jitter_factor: f64) -> Duration {
    if jitter_factor == 0.0 {
        return base;
    }

    let base_secs = base.as_secs_f64();
    let min_secs = base_secs * (1.0 - jitter_factor);
    let max_secs = base_secs * (1.0 + jitter_factor);

    // Clamp min to 0 (jitter_factor = 1.0 would give min = 0.0)
    let min_secs = if min_secs < 0.0 { 0.0 } else { min_secs };

    let jittered = rand::thread_rng().gen_range(min_secs..=max_secs);
    Duration::from_secs_f64(jittered)
}
```

### Implementation: RetryExecutor

**File:** `src/retry/mod.rs`

Change `compute_delay()` to accept jitter_factor and apply jitter after computing the exponential backoff:

```rust
fn compute_delay(config: &RetryConfig, attempt: u32) -> Duration {
    // ... existing exponential backoff math ...
    let base_delay = Duration::from_secs_f64(clamped);
    crate::jitter::apply_jitter(base_delay, config.jitter_factor)
}
```

### Implementation: RateLimiter

**File:** `src/rate_limiter/mod.rs`

In `acquire_permission()`, after computing `sleep_until` (the next window boundary), apply jitter to the remaining wait time:

```rust
// Current: tokio::time::sleep_until(sleep_until).await;
// New:
let remaining = sleep_until.duration_since(Instant::now());
let jittered = crate::jitter::apply_jitter(remaining, config.jitter_factor);
tokio::time::sleep(jittered).await;
```

Note: this means the sleep may wake up slightly before or after the window boundary. That's intentional — the next loop iteration will check permits again.

### Implementation: CircuitBreaker

**File:** `src/circuit_breaker/mod.rs`

When the circuit breaker transitions to half-open state, it waits `wait_duration_in_open_state`. Apply jitter to this wait:

The circuit breaker currently tracks `opened_at` timestamp and computes whether enough time has passed. The jitter needs to be computed once at open time (not rechecked on every `check_permitted` call, which would give different answers each time).

Add a `jittered_wait` field to the per-key state that is set once when the circuit opens:

```rust
struct KeyState {
    // ... existing fields ...
    /// Actual wait duration after jitter (set once when circuit opens).
    jittered_wait: Duration,
}
```

When transitioning to Open state:
```rust
entry.jittered_wait = crate::jitter::apply_jitter(
    config.wait_duration_in_open_state,
    config.jitter_factor,
);
```

When checking if half-open is ready:
```rust
// Current: elapsed >= config.wait_duration_in_open_state
// New:     elapsed >= entry.jittered_wait
```

## Backward Compatibility

- `jitter_factor` defaults to `0.0` everywhere → existing behavior unchanged
- `#[serde(default)]` on the field → existing serialized configs deserialize correctly
- No changes to trait signatures → all existing implementations and callers work unchanged
- New dependency (`rand`) is the only breaking change at the Cargo level

## Testing Strategy

1. **Unit test `apply_jitter()`**: verify bounds (output always in `[base*(1-j), base*(1+j)]`), verify 0.0 factor returns exact base, verify 1.0 factor can return near-zero
2. **Statistical test**: call `apply_jitter()` 1000 times, verify the distribution covers the expected range (not clustered at one end)
3. **RetryExecutor test**: with jitter_factor=0.5, run 10 retries, verify all delays are within expected bounds but not all identical
4. **RateLimiter test**: with jitter, verify permits are still correctly rate-limited (jitter doesn't break the permit counting)
5. **CircuitBreaker test**: with jitter, verify the circuit still transitions through Closed→Open→HalfOpen→Closed, just with variable timing
6. **Backward compat test**: configs without jitter_factor deserialize correctly and produce deterministic delays (same as before)
